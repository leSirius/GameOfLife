# 性能优化
计算量来源：矩阵计算；矩阵拷贝成本；
绘制效率比较高
在稀疏矩阵上，矩阵拷贝成本不可忽略（在template配合1000x1000扩展地图，这两个成本比较接近）；在稠密矩阵上，性能开销主要在矩阵计算，计算量与活跃点数（或者邻居map）的规模成线性关系，而1000x1000的复制基本1ms以内，

1. 合并循环。每次执行每次重新计算状态并绘制时，从三个循环合并成两个或一个。这个优化没有明显效果，合并循环有可能提升缓存命中率，但几乎不会节省计算量。
2. 移除react渲染。在一个小测试集上，每次update执行的脚本时间达到了10毫秒左右，经过排查原因是setState导致了react渲染。特别是，左侧控制栏上方的几个input和button组合（这里使用了NextUi库的组件），使用useMemo和memo优化（或者删除）这些组件可以使脚本时间减少到3毫秒左右。但把setState改为使用useRef追踪状态更好一些，这样用时0.5ms左右。
3. 并行。这个游戏的矩阵计算除了邻居以外不存在数据依赖，比较适合并行优化。但因为worker线程不能与主线程共享内存，切分、合并矩阵，收发消息都导致了额外的成本，这让该版本在稀疏矩阵或者活跃点较少时没有明显优势（template只有100x100）。
当矩阵比较稠密或者活跃点较多时，并行计算在性能上显现出优势。在500x500，稠密度0.5的地图上，并行版本主线程执行时间在80-90ms之间，worker线程大约30ms左右。而串行版本大约180-200ms之间 （这里的比较对象是一个针对稀疏矩阵减少过矩阵复制的串行版本，不过在稠密情况下，性能受到矩阵复制影响不明显。因为用的随机排布，测试变化幅度较大，但该参数下并行版本主线程基本100ms以内，串行性能得100ms开外）。
不过因为worker线程的消息处理是在主线程提交绘制之后进行的，所以并行版本绘制时间明显多一些。
4. 稀疏矩阵。在稀疏矩阵上，矩阵的复制成本相对于活跃点计算变得难以忽视。在使用template但扩容地图到1000x1000时，遍历的串行版本需要10ms左右，经过liveList记录活跃点列表与使用map记录邻居数量，优化版本用时在2到3毫秒左右，而创建一个空的1000x1000矩阵或者同规模深拷贝大约0.5-1ms左右，原本useRef使用了深拷贝，在这里不稳妥地改为了浅拷贝，减少了一次调用。

可以继续考虑的优化点：深拷贝写时复制，整个游戏地图使用活跃点列表存储或者并行加稀疏优化，离屏canvas，极端情况下的线程安全

# SSR与动态导入
## 需求
canvas的绘制需要`window.devicePixelRatio`条件样式宽度与宽度属性的比例，该值只可以在浏览器获取，这与SSR冲突。使用动态导入可以解决这个问题，Next.js 提供了`dynamic`函数。在服务端通过处理`undefined`设置默认值1以后，浏览器显示canvas的情况可以反映当前呈现的是哪里的绘制结果。
测试中的父组件是page组件，子组件是GameOfLife组件，页面主要元素在子组件。为了禁用SSR可以使用动态导入，这个子组件是初始渲染必须的，所以动态加载节省初始渲染成本的用途不适用。

更合理的应当是，把使用`window.devicePixelRatio`的state放到`canvas.js`组件中，这样侧边栏可以被预渲染，然后GameOfLife组件使用另一个state来实现原本控制放大的功能。调节样式宽度与canvas宽度属性的比例会导致模糊，使用`GridSize`其实更好一些。

page.js     ->     GameOfLife.js     ->    canvas.js
 
                                     ->    nav-bar.js

测试的一些设置不具有合理性，只是观察或猜测一下关于SSR、模块以及浏览器渲染的其他相关行为。

## 动态导入边界
动态导入意味着需求时响应（发起请求，打包或者执行），不总是单独下载的。只有客户端组件会被动态导入，即使动态导入一个服务端组件，也只有它的客户端子组件是动态导入的。

动态导出是一个瞎起的名字，指在game-of-life文件内动态导入GameOfLife组件（返回一个期约，会导致该组件的获取进入微队列，其渲染顺序靠后），然后导出这个动态导入的结果。父组件导入的时候则是静态的，所以子组件中的其他代码包括静态导入文件，如果有执行语句或调用的话会被执行。

两者区别或可看作是按需执行边界的位置不同。

## 使用静态导入：
1. 通常浏览器呈现的是服务端预渲染的结果。浏览器端可以打印出1.75的正确缩放比；报错服务端与客户端渲染结果（canvas的样式宽度）不匹配。本机测试时，刷新页面开启节流时才会察觉闪烁（除了蓝色背景外所有的内容消失重绘一下）。

浏览器收到的一个预渲染完成的html，然后客户端react组件在浏览器执行并水合。react-dom会报错提示渲染服务端与服务端渲染结果不同。因为页面是直接解析html渲染的，不用等待JS加载与执行，这种呈现方式是最快的，所以刷新闪烁最短。
（虚拟DOM的比较和选择是怎么做的？浏览器白屏时间是在什么时候？）

2. 当在子组件中未处理`undefined`就使用依赖浏览器环境的值时，浏览器会呈现本地渲染的页面结果，刷新会闪白屏。错误码`500`。

在打印window对象等依赖浏览器的变量时，子组件的预渲染因为undefined导致渲染失败，收到的html是空的（连蓝色背景都没有），服务端会报错渲染失败的错误。页面内容全由js加载和解析提供，所以空白时间明显增加。服务端环境与客户端对undefined的处理不同。

## 当使用动态导入
### 显式禁用ssr，浏览器总是得到一个几乎为空的HTML，初始呈现速度较SSR慢：

1. 如果父组件是客户端组件，导入方式为在父组件动态导入。GameOfLife在一个单独的js文件中，对这个js的请求由page.js发起，不能被preload scanner优化。

浏览器收到html文件，从`<header>`里请求page.js文件（async属性），然后等page.js加载完执行时请求game-of-life组件文件（初始渲染即需要）。请求`<header>`内的文件时，浏览器有优化，preload scanner会继续解析后面的请求，并几乎同时发起它们（包括page.js)。然而，page.js文件内的请求无法被检测到，因而无法优化，所以请求很迟，请求发起时间在600ms-800ms之间，比ico文件还晚（ico文件是在`<header>`中的，应该与page.js同时探测到，它加载特别晚应该是因为优先级特别低）。
（这种情况中，page.js是在需要调用子组件时才发起请求，否则不会发请求，所以优势在懒加载）

2. 如果父组件是客户端组件，导入方式为在子组件动态导出。GameOfLife组件打包进page.js，没有分包传输。

在这种导入方式中，父组件是静态导入的子组件，所以子组件被直接打包进了page.js，当page.js执行到相应位置的时候可以从本文件获取导入的组件，不再发起请求。在只需要禁用SSR的需求下，这个方法比上一个效率高。

3. 如果父组件是服务端组件，导入方式为在子组件动态导出。GameOfLife组件打包进page.js，没有分包传输。

4. 如果父组件是服务端组件，导入方式为在父组件动态导入。GameOfLife组件打包进page.js，没有分包传输。

父组件是服务端组件的情况下，浏览器上的执行情况没有发现或想到明显区别。请求到的html文件有些轻微不同，page.js文件也有。

- 服务端组件的动态导入即使没有调用也会打包到page.js中，所以它似乎会打包导入的所有的客户端js代码。
- 客户端父组件静态导入，子组件会被打包进page.js，动态导入则不会。

### 当使用动态导入，启用SSR时，浏览器得到一个预渲染的HTML：
在这里html冲突总是存在的，然而为什么这里有3种被改为了客户端版本？与静态导入时的状况有什么不同，特别是父组件同为静态导入时，子组件导出不同为什么会有影响？为什么服务端组件动态导入的那个没改的？

1. 父组件是客户端组件，导入方式为父组件动态导入时。页面先显示预渲染的HTML页面尺寸，canvas没有绘制，然后变成浏览器渲染页面，除了背景以外的元素都消失然后重绘。GameOfLife组件被分包传输。

因为`<header>`中的所有脚本加载都是`async`的，并不会阻塞html解析，所以浏览器呈现了SSR的结果。绘制canvas的js都还没加载。加载执行page.js后开始下载game-of-life.js文件，然后子组件在客户端渲染修改了页面。这里的情况有些类似于 FOUC。
这里会感受到消失重绘，是因为page.js在执行时，没有子组件代码，只能提交请求，然后完成渲染提交了结果，导致页面元素消失。实际上，对子组件代码的请求发起很迟，在OnLoad事件之后由一段react-dom代码中的`mountLazyComponent`发起，这段react-dom代码之后的提交让屏幕的元素消失。随后子组件加载完成并执行，再次修改页面。

2. 父组件是客户端组件，导入方式为子组件动态导出时。页面先显示预渲染的HTML页面，canvas没有绘制，然后变成浏览器渲染页面，不会感觉到消失。

与之前类似，子组件不分包，只是收到的html中有了预渲染的结果。也是类似的，对于初始渲染，这种方法比`1.`体验好和速度快

3. 父组件是服务端组件，导入方式为子组件动态导出时。页面先显示预渲染的HTML页面，但canvas没有绘制，然后变成浏览器渲染的页面。

4. 父组件是服务端组件，导入方式为父组件动态导入时。页面显示预渲染的HTML，不再发生变化。
